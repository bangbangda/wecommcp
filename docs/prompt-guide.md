# Tool Description 与 System Prompt 编写规范

> Tool Description 决定 LLM「什么时候调、怎么调」工具。
> System Prompt 决定 LLM「怎么思考、怎么交互」。
> 两者协同决定了 AI 助手的智能水平。

---

## 一、Tool Description 编写规范

### 核心原则

Tool Description 是 LLM 做决策的第一依据。LLM 看到用户消息后，先扫描所有 Tool 的 Description，判断该调哪个。Description 写得好不好，直接决定了工具调用的准确率。

### 必须包含的四个要素

1. **做什么** — 核心功能，一句话说清
2. **什么时候用** — 触发场景 + 用户可能的自然语言表达示例
3. **怎么配合** — 和其他工具的前置/后置依赖关系
4. **不能做什么** — 明确边界，避免误调用

### 项目实际示例

#### 好的 Description

```php
// QueryMeetingsTool — 完整覆盖四要素
#[Description(
    '查询用户的会议列表。' .                                          // ① 做什么
    '当用户想取消、修改、查看会议但没有提供明确的会议ID或标题时，' .       // ② 什么时候用
    '应先调用此工具获取会议列表。' .                                    // ③ 怎么配合
    '支持按创建时间、开始时间排序，支持关键词搜索。' .
    '典型场景："我今天有什么会""取消我刚刚创建的会""上午那个会是几点"'  // 泛化示例
)]

// CancelMeetingTool — 写明调用链 + 泛化示例
#[Description(
    '取消企业微信会议。' .
    '优先通过 meetingid 精确定位，' .
    '如果用户没有提供明确的会议标识（如说"取消刚才那个会"），' .         // 口语化泛化示例
    '应先使用 query_meetings 工具查询会议列表获取 meeting_id，' .      // 前置依赖
    '并向用户确认后再调用此工具。' .
    '这是一个危险操作，Tool 会返回会议信息供确认，而非直接删除。'        // 边界说明
)]
```

#### 坏的 Description

```php
// ❌ 太简单，LLM 不知道什么时候该调
#[Description('查询会议')]

// ❌ 太技术化，缺少场景描述
#[Description('调用企微 API /cgi-bin/meeting/list 获取会议列表')]

// ❌ 没有说和其他工具的关系
#[Description('查询用户的会议列表，支持按时间和关键词筛选')]
```

### 关键技巧

#### 技巧1：写明工具间的调用链

当工具 B 依赖工具 A 的输出时，在工具 B 的 Description 中明确写出。这是触发 LLM 链式调用的关键：

```php
// GetMeetingInfoTool — 明确前置依赖
#[Description(
    '查询企业微信会议详情。' .
    '优先通过 meetingid 精确定位，' .
    '如果用户没有提供明确的会议标识（如说"查一下刚才那个会"），' .
    '应先使用 query_meetings 工具查询会议列表获取 meeting_id，' .  // ← 引导链式调用
    '再调用此工具获取完整详情。'
)]
```

#### 技巧2：用泛化示例覆盖口语表达

不需要穷举所有表达，给 2-3 个典型示例即可，LLM 会自动泛化：

```php
// ✅ 给典型场景，LLM 自动理解变体
'典型场景："我今天有什么会""取消我刚刚创建的会""上午那个会是几点"'

// ❌ 试图穷举所有表达（永远列不完）
'当用户说"查会议"或"看看会议"或"有什么会"或"会议列表"或...'
```

#### 技巧3：区分相似工具的边界

项目中 `query_meetings` 和 `get_meeting_info` 容易混淆，Description 要明确各自职责：

```php
// query_meetings — 列表查询，轻量
'查询用户的会议列表...支持关键词搜索'

// get_meeting_info — 单个会议的完整详情，需要 meetingid
'查询企业微信会议详情...优先通过 meetingid 精确定位'
```

### Schema 参数 Description 规范

参数描述决定了 LLM 提取参数的准确性：

```php
// ❌ 差的参数描述
'start_time' => $schema->string('开始时间')

// ✅ 好的参数描述：写明格式、给出示例
'start_time' => $schema->string('开始时间，ISO 8601 格式，例如 2026-02-25T14:00:00')

// ❌ 差的参数描述
'invitees' => $schema->array('参会人', items: $schema->string())

// ✅ 好的参数描述：写明内容要求和处理方式
'invitees' => $schema->array(
    '其他参会人的中文姓名列表，不需要包含当前用户（会自动作为管理员加入）',
    items: $schema->string('参会人姓名')
)
```

## 二、System Prompt 编写规范

### 结构模板

System Prompt 在 `ChatService::buildSystemPrompt()` 中维护，按以下顺序组织：

```
┌──────────────────────────────────────────┐
│ 1. 身份定义       — 你是谁                │
│ 2. 环境上下文     — 当前时间、当前用户      │
│ 3. 能力列表       — 从 ToolRegistry 自动生成│
│ 4. 注意事项       — 通用行为规则           │
│ 5. 操作规范       — 具体场景的执行步骤      │
│ 6. 用户记忆       — 关于这个用户的偏好（未来）│
└──────────────────────────────────────────┘
```

### 各部分编写指南

#### 1. 身份定义（简洁，一句话）

```
你是企业微信 AI 助手，帮助用户管理会议和查询联系人。
```

不要写过长的身份描述，LLM 不需要背景故事。

#### 2. 环境上下文（动态注入，每次请求更新）

```
当前时间：2026-02-26 18:30:00 (Wednesday)
当前用户：zhangsan
```

必须包含当前时间，否则 LLM 无法处理"今天""明天""下周"等时间表达。包含星期几有助于处理"这周五""下周一"。

#### 3. 能力列表（自动生成，无需手动维护）

由 `ToolRegistry::getCapabilitiesSummary()` 从各 Tool 的 `#[Description]` 自动提取，保持与 Tool Description 一致。

#### 4. 注意事项（正向指令优先）

用"应该做什么"而非"不要做什么"。正向指令比否定指令效果好得多：

```
❌ "不要在参数不完整时瞎猜"
✅ "参数不足时主动追问用户补充"

❌ "不要在用户没确认的情况下删除会议"
✅ "取消会议前，展示会议信息并等待用户确认"
```

#### 5. 操作规范（最重要，引导链式推理）

对关键场景给出明确的执行步骤，减少 AI 自由发挥空间：

```
会议查询/取消操作规范（重要）：
- 除非用户说出了完整且唯一的会议标题，否则必须先调用 query_meetings 获取列表
- query_meetings 支持模糊搜索，"周会"可匹配"产品周会""技术周会"，充分利用
- 拿到列表后以编号形式展示，让用户选择编号，而不是要求用户重新描述
- 用户回复编号后，使用对应会议的 meetingid 调用后续工具
```

**链式推理是核心能力。** 当用户指令模糊时，AI 应主动分解任务，通过多步工具调用完成：

```
用户: "取消下午的会"
→ 步骤1: 调用 query_meetings(time_range: "today") 查询今天的会议
→ 步骤2: 筛选下午时段的会议，以编号列表展示给用户
→ 步骤3: 用户选择后，调用 cancel_meeting(meetingid: xxx) 执行取消

用户: "帮我建个明天的会，叫上小王和小李"
→ 步骤1: 调用 search_contacts(name: "小王") 搜索联系人
→ 步骤2: 调用 search_contacts(name: "小李") 搜索联系人
→ 步骤3: 如有同音字多候选，向用户确认
→ 步骤4: 调用 create_meeting(title: ..., invitees: [...]) 创建会议
```

### System Prompt 反模式

```
❌ 过度啰嗦
"你是一个非常智能的、善解人意的企业微信AI助手..."
→ 浪费 tokens，LLM 不需要形容词

✅ 简洁直接
"你是企业微信AI助手，帮助用户管理会议和查询联系人。"


❌ 穷举用户表达
"当用户说'建个会'或'开会'或'安排会议'或'约个会'或..."
→ 永远列不完，LLM 本身能理解同义表达

✅ 给典型场景，信任 LLM 泛化能力
"典型场景：'建个明天10点的会讨论产品需求'"


❌ 混入技术实现细节
"调用 /cgi-bin/meeting/create 接口时需要传入 access_token..."
→ 这是 Service 层的事，不该出现在 prompt 里

✅ 只描述业务行为
"创建会议时需要主题和开始时间，参会人可选。"
```

## 三、Tool Description 与 System Prompt 的协同

两者各有分工，不要重复也不要矛盾：

| 职责 | Tool Description | System Prompt |
|------|-----------------|---------------|
| 工具选择 | ✅ 主要依据 | 辅助引导 |
| 调用链 | ✅ 写明前置依赖 | ✅ 给出完整步骤 |
| 参数格式 | ✅ Schema 定义 | 补充通用规则（如时间格式） |
| 交互方式 | — | ✅ 主要职责（确认、追问、列表选择） |
| 安全约束 | 边界说明 | ✅ 主要职责 |

**原则：Tool Description 管"能做什么"，System Prompt 管"怎么做"。**

## 四、迭代优化流程

提示词不是一次写完的，需要持续优化：

```
1. 使用中发现问题
   例："用户说'取消下午的会'，LLM 直接调了 cancel_meeting 而没有先查询"

2. 诊断原因
   → cancel_meeting 的 Description 里有没有写明需要先查询？
   → System Prompt 里有没有链式推理的引导？
   → 还是 Schema 参数设计的问题？

3. 最小化修改
   → 只改出问题的那个 Description 或 Prompt 片段
   → 不要大规模重写，避免引入新问题

4. 验证效果
   → 用相同场景测试是否修复
   → 同时测试相关场景是否受影响
```

建议在代码注释中记录 prompt 修改原因：

```php
// 2026-02-26: Description 加入"应先使用 query_meetings"引导
// 原因：用户说"取消刚才那个会"时 LLM 直接调 cancel_meeting 导致失败
#[Description('取消企业微信会议。优先通过 meetingid 精确定位...')]
```

## 五、Token 预算参考

```
System Prompt:
  身份 + 环境上下文         ~150 tokens   动态生成
  能力列表（5 个工具）      ~300 tokens   自动生成
  注意事项 + 操作规范       ~400 tokens   手动维护
  ─────────────────────────
  小计                     ~850 tokens

Tool 定义:
  每个 Tool               200-400 tokens
  5 个 Tool              1,000-2,000 tokens

对话历史:
  每轮（含 tool_use）     300-800 tokens
  保留 20 条消息         3,000-8,000 tokens

─────────────────────────────────
总输入预算:           ~5,000-11,000 tokens
```

当前阶段（5 个工具、Ollama qwen3:8b 8K context / Claude 200K context）预算充裕，无需优化。未来工具数量超过 20 个时，可考虑：按需加载 Tool → 压缩对话历史 → 精简推理示例。
